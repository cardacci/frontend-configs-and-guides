## Overview

AI coding agents like Claude can generate better, more consistent code when they have
access to your project's coding standards. By placing standard documents inside a
`.claude` folder at the root of your project with a `CLAUDE.md` index file, the agent
automatically reads and follows those rules when generating or reviewing code.

This approach ensures that **every component generated by the AI** follows the same
structure, naming conventions, and ordering rules that your team has agreed upon.

---

## What is the `.claude` Folder?

The `.claude` directory is a convention recognized by Claude-based tools (such as
Claude Code and IDE integrations) as a source of **project-specific instructions**.

| File / Folder | Purpose |
|---|---|
| `.claude/CLAUDE.md` | Index file â€” the main instruction document the agent reads first |
| `.claude/standards/` | Folder containing detailed standard files referenced by `CLAUDE.md` |

When Claude starts a coding session in your project, it looks for `CLAUDE.md` at the
project root **or** inside the `.claude` directory and applies its contents as context
for all code generation tasks.

## Project Setup

### Step 1 â€” Create the Directory Structure

At the root of your project, create the `.claude` folder with a `standards`
subdirectory:

```bash
mkdir -p .claude/standards
```

### Step 2 â€” Copy the Standard Files

Copy the component structure guides into the standards folder. Choose the files that
match your project:

**TypeScript project:**
```bash
cp path/to/TSX_components.md .claude/standards/TSX_components.md
```

**JavaScript project:**
```bash
cp path/to/JSX_components.md .claude/standards/JSX_components.md
```

**Mixed project (both):**
```bash
cp path/to/TSX_components.md .claude/standards/TSX_components.md
cp path/to/JSX_components.md .claude/standards/JSX_components.md
```

## CLAUDE.md Template

### For TypeScript (TSX) Projects

```markdown
# Project Coding Standards

## React Components (TSX)

When generating, modifying, or reviewing React TypeScript components, follow the
structure and conventions defined in:

â†’ [TSX Components Structure Guide](standards/TSX_components.md)

### Key Rules Summary

- **Component structure order:** Imports â†’ Types/Interfaces/Enums â†’ Constants â†’
  Component Function â†’ Redux â†’ Hooks â†’ State â†’ Refs â†’ Derived State â†’ Callbacks â†’
  Effects â†’ Functions â†’ Render Helpers â†’ JSX Return â†’ Exports.
- **Alphabetical ordering:** Sort everything that can be sorted alphabetically within
  its block (props, state, refs, callbacks, effects, imports, exports, interface
  properties, enum values).
- **Types outside components:** All `type`, `interface`, and `enum` definitions must be
  at module scope, never inside the component function body.
- **Props interface last:** The component's `Props` interface should be the last
  interface defined before the component function.
- **Naming conventions:**
  - `handle*` for event handlers defined in the component.
  - `on*` for event handler props passed from parents.
  - `is*` / `has*` for boolean state and props.
  - `get*` for data computation functions.
  - `render*` for functions returning JSX.
- **Imports grouped:** External libraries â†’ Internal hooks â†’ Components â†’ Types â†’ Utils.
```

## Tips & Best Practices

| Tip | Detail |
|---|---|
| **Keep `CLAUDE.md` concise** | The index file should summarize rules and reference detailed docs. Long instructions risk being truncated. |
| **Link, don't duplicate** | Reference the standard files instead of copying their full content into `CLAUDE.md`. |
| **Version your standards** | Keep the `.claude` folder in version control so the whole team (and the AI) uses the same rules. |
| **Update as standards evolve** | When your team updates a coding convention, update the files in `.claude/standards/` accordingly. |
| **Test the output** | After setting up, ask Claude to generate a sample component and verify it follows the structure. |
| **Combine with other standards** | You can add more files to `.claude/standards/` for other conventions (e.g., API patterns, testing standards). |
| **Root-level `CLAUDE.md`** | Alternatively, you can place `CLAUDE.md` directly at the project root instead of inside `.claude/`. Both locations are supported. |
